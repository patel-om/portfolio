No response after accepted address axi pack v1.0.0 · usually owned by design (RTL)
Pattern match100%

An address handshake completed but no response beat ever arrived; the run timed out with the transaction still outstanding.

Typical causes:

  • Response FSM stuck (missing arc back to a ready state)
  • Outstanding-transaction counter overflow or leak
  • Response multiplexer never selects the completing source
  • Credit/token starvation between clock domains

Signals to inspect: ARVALID, ARREADY, RVALID, RREADY, RLAST

Matched evidence: timeout observed (ev-8107973e67ec, ev-d61d8c4f98f1); transaction outstanding (ev-50f542e1219c, ev-6b9e3dd65182, ev-f0d089afd5a4, ev-8107973e67ec); AXI context (ev-50f542e1219c, ev-f0d089afd5a4)

References: AMBA AXI Protocol Specification §A3.4.1 (Read response dependency: RVALID must eventually follow an accepted AR.)

Debug playbook: AXI read timeout (7 deterministic steps)
  1. Check ARVALID at the master boundary Confirm the master actually issued the read.signals: ARVALID
  2. Check ARREADY from the slave Did the address handshake complete, or is the slave back-pressuring forever?signals: ARREADY

Verification concepts in play

Outstanding transactions (axi pack) · Accepted addresses without completed responses are outstanding. A DUT that accepts addresses but never responds deadlocks the master; outstanding counters are the first thing to check on an AXI hang.

AXI read transaction (axi pack) · A read is an AR-channel address handshake followed by one or more R-channel data beats; the transaction stays outstanding until the last beat (RLAST) is accepted.

VALID/READY handshake (axi pack) · Every AXI channel transfers when VALID and READY are high in the same cycle. A source must keep VALID asserted (with stable payload) until the handshake completes; dropping VALID early is a protocol violation.
